# topogram-demo-todo

Generated by `topogram new --template todo` from the Topograms catalog. The
catalog resolves `todo` to `@attebury/topogram-template-todo@0.1.29`.

## Workflow

Create a fresh Todo app from the catalog with:

```bash
topogram template list
topogram catalog show todo
topogram new ./todo-demo --template todo
cd ./todo-demo
npm install
npm run doctor
npm run check
npm run generate
npm run verify
```

This repository is the maintained generated consumer for that flow. Day-to-day
checks here are:

```bash
npm install
npm run explain
npm run doctor
npm run check
npm run catalog:template-list
npm run catalog:show
npm run template:policy:check
npm run template:status:latest
npm run template:update:recommend
npm run generate
npm run verify
```

Edit `topogram/` and `topogram.project.json`, then regenerate with `npm run generate`.
Generated app code is written to `app/`.

This repo owns generated Todo app verification as an external package consumer.
The main Demo Verification workflow runs Topogram checks, template policy/update
checks, generated app compile, and the generated runtime command against a
Postgres service.

See [`CONSUMER_PROOF.md`](./CONSUMER_PROOF.md) for the verification standard
this repo must keep before it counts as external consumer proof.

Runtime checks intentionally exercise Todo template semantics. CI uses a
manager/admin bearer-demo identity for manager/admin endpoints such as export
read/download/delete, while generated web checks still verify user-scoped app
behavior.

## Updating Topogram CLI

Use the Topogram CLI command so `package.json` and `package-lock.json` are
updated from published GitHub Packages registry metadata:

```bash
NODE_AUTH_TOKEN=<github-token-with-package-read> npm run update:topogram-cli -- --latest
```

The command resolves the latest published CLI package, installs
`@attebury/topogram@^<version>`, refreshes stale lockfile tarball metadata when
needed, updates `topogram-cli.version`, then runs `cli:surface`, `doctor`,
`catalog:show`, and `check`. Do not hand-edit `package-lock.json` tarball URLs
or integrity values. After pushing, the normal Demo Verification workflow is
the package-consumer compile/runtime gate.

## Detaching Template Metadata

The Todo demo keeps template metadata so it can verify template update behavior.
For a maintained app that should fully own its local files, first review the
plan with:

```bash
npm run template:detach:dry-run
```

Then run `npm run template:detach` when the project should stop tracking
template update metadata. Executable implementation trust is retained when
`implementation/` remains configured.
Use `npm run source:status` for local provenance, trust, and baseline checks.
Use `npm run source:status:remote` only when you intentionally want package
registry status.
